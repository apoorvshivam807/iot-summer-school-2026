# Q20. (Theory) Arduino Core Lifecycle Functions & Blocking vs Non-Blocking Execution

This document details the operational mechanics of the primary execution blocks in Arduino, analyzes the negative system-wide performance degradation caused by blocking functions like `delay()`, and outlines the industry-standard non-blocking design alternative.

---

## 1. Core Functions Explanation

### void setup()
* **Execution Profile:** Runs exactly **once** when the microcontroller completes its hardware initialization sequence following a physical power-on, reset button strike, or downstream bootloader execution trigger.
* **Primary Objective:** Configures the initial system landscape. This includes designating GPIO pin directions via `pinMode()`, configuring baud rates to establish communication pipelines via `Serial.begin()`, and calling address registration handshakes to prepare peripheral networks ($I^2C$ / $SPI$).

### void loop()
* **Execution Profile:** Runs **continuously** and infinitely in a sequential cycle. The absolute moment the closing bracket of the `loop()` function is processed, the underlying hardware instruction registers point straight back to its opening line to run the sequence again.
* **Primary Objective:** Manages the active real-time operational logic of the embedded asset, including checking sensor pins, processing analytical control loops, handling communication arrays, and driving dynamic hardware actuators.

---

## 2. The Mechanics and Impact of a Long delay()

The standard `delay(ms)` command executes a **blocking, busy-wait loop** directly within the processor core. When executed, it instructs the internal clock system to run empty loop cycles until the requested millisecond counter constraint is completely exhausted.
