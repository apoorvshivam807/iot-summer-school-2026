## Q9. (Theory) Explain the difference between `git pull`, `git fetch`, and `git clone`. When would you use each? (2 marks)

### 1. `git clone`

* **What it does:** Downloads an existing remote repository (from GitHub) and creates an exact, full local copy of it on your computer. It automatically initializes the `.git` directory, sets up tracking branches, and links the remote reference (`origin`).
* **When to use it:** Use this when you are starting to work on a project for the **very first time** on a machine and need a local copy of the codebase.

---

### 2. `git fetch`

* **What it does:** Downloads the latest commits, files, and branch updates from the remote repository to your local system, but it **does not merge** or change your active local working files. It only updates your hidden remote-tracking repository branches (e.g., `origin/main`).
* **When to use it:** Use this when you want to see what your teammates have worked on and review remote changes **before safely integrating them** into your code.

---

### 3. `git pull`

* **What it does:** Downloads the latest changes from the remote repository and **instantly forces an automatic merge** into your active local working branch. It is a combined shortcut operation:

$$\text{git pull} = \text{git fetch} + \text{git merge}$$

* **When to use it:** Use this when you want to quickly **synchronize and update** your local repository to match the latest state of the remote repository on GitHub before you start writing new code.

---

### 📊 Summary Reference Table

| Command | Modifies Local Workspace? | Downloads Remote Data? | Typical Frequency |
| --- | --- | --- | --- |
| **`git clone`** | **Yes** (Creates everything) | Yes (Downloads all history) | Only once per project machine setup |
| **`git fetch`** | **No** (Safe inspection view) | Yes (Updates tracking index) | Frequently (To peek at remote changes) |
| **`git pull`** | **Yes** (Updates active files) | Yes (Downloads and merges) | Daily (Before starting active code work) |
